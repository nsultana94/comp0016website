Implementation

Building our two platforms from ground up

Overview of the System Implementations


The SharePoint Platform

Implementing the design in SharePoint was a challenging process as our team were not familiar with the user interface of the design page, and the multiple Power Apps that were available to us. However, due to the extensive research and prototyping that had been done, we began to learn the ropes of the SharePoint interface and after a number of weeks, the majority of the functionalities were implemented as intended.

Once a set number of platforms have been identified, the next steps were to make a note of each platform, categorising the functionalities and overall design of their system, looking at the pros and cons, hence providing us with more information on how we would like to build our project proposal engine.

The features will be outlined below in-depth.

The Django Platform

We designed the open-source system using the Django web framework for back-end, which heavily relies on Python as the core programming language.

We felt more comfortable coding in Python, nevertheless, we took into consideration a wider gamut of programming languages and wisely compared their key features. Despite the fact that Python is dynamically typed and required more testing due to potential runtime errors, it comes with extensive libraries and modules that covered our needs in web development. However, we encountered some initial hurdles when using Django because we needed to learn it from scratch. The process of selecting tutorials, of self-teaching, and of putting the theory into practice was thought-provoking and, indeed, challenging.

As regards to the front-end development, we decided to employ Bootstrap 4, a free and open-source framework. Our choice was backed up by its exhaustive assembly comprised of CSS, HTML, JavaScript.

We have successfully deployed our web portal using the SQLite3 Database. We opted for this relational database, given that it is part of the Python distribution. Also, when building up the skeleton of the project, Django automatically generated a reliable setup for an SQLite3 back-end database.

Submission Portal and display of projects

 
SharePoint
The submission portal was implemented using the exisiting functionality of Microsoft Forms. The form was separated into sections for a more user-friendly experience. The form is connected to a list within the SharePoint site, which will automatically add the data of the new responses to the list using Microsoft Power Automate.

The data is then presented catagorically in a Microsoft List which can then be filtered, and queried with the inbuilt features within Microsoft Lists. Data in the fields of are visually represented using pills, and a range of different views based on the need of the reviewer.

Fig. Microsoft Power Automate flow for project proposals

The image above shows the flow of data in the submission of projects. When a project is submitted, this acts as a trigger for this flow which obtains the response details and adds it to the appropriate project submissions list with the correct data and adds it to the excel sheet storing the project proposals.

Django

The submission portal in Django was implemented using in-built Django forms which allowed us to access the fields required from the Person model. Below there is the Person table with all the requested information for both the submitter and the details specific to the project proposal. All the Django models are classes implemented as subclasses of the django.db.models.Model and will therefore be able to integrate within them various fields, methods and metadata. 

my_field_name = models.CharField(max_length=20, blank=True, default="", verbose_name='Enter field info')

Our above example is a field called my_field_name, of type models.CharField — which means that this field will contain strings of alphanumeric characters. The field types are assigned using specific classes, which determine the type of record that is used to store the data in the database, along with validation criteria to be used when values are received from an HTML form (i.e. what constitutes a valid value). The field types can also take arguments that further specify how the field is stored or can be used.

In the Person model we defined fields such as name, project_title, summarised_abstract, department, organisation etc. of type models.CharField, but we added fields of types models.TextField that allowed us to define a larger text area spanning on multiple rows and columns, of type models.EmailField for validating emails, models.URLField for GIT links, or models.ChoiceField for the data which had to display to the user a series of options to choose from.

Reviewing Projects and giving feedback


SharePoint

Each user can be assigned to a project using the inbuilt Microsoft 365 feature in SharePoint, where they can then give feedback by clicking a button which will trigger a flow that will prompt the user for inputs such as the feedback they wish to give, the RAG rating, and a dropdown multi list option of extra feedback details. These feedback details will then be appended to the feedback columns in the proposal submissions list, and the number of red/amber/green ratings will be incremented based on the reviewer's answer.

This will then be used to update the specific project in the SharePoint list as well as the excel sheet to reflect the newly added changes.

The list column is formatted using json to only show when a project has not been accepted / rejected and clicking on the button will trigger the flow, and only members within the site organisation who can edit the site are able to trigger the flow.

Only the users with the authenticated permission of a reviewer can have access to the Review Portal. An IXN Lead or the so-called approver is able to assign reviewers on the page of every specific projects. The assignment is done in back-end through a POST request that allows the selection from a drop-down menu from among the reviewers, that are stored as a Many-to-Many field in the Person model. We also added fields of types models.IntegerField that allowed us to define the scores the reviewer can give for the 7 criteria. Similarly, the saving is done automatically with the load and submission of the review form and the fields self.points will store the summation of the seven scores, up to a total of 35, and self.score will convert this sum on a scale from 0 to 100, through the rule of three.

Approval of Projects


SharePoint

Another button is displayed in the sharepoint list, if the status of the project is neither accepted nor rejected. This button can be triggered by any reviewer once all feedback has been sent, to send an email to the approver of the project, giving details of the project, the feedback, the RAG ratings and list of reviewers. This starts a Microsoft Approval process, where the approver can accept or reject the project through that and this will update the status accordingly based on the answer. If the project is accepted, a new item will be created in the Accepted Projects list and the submitter will be informed via email.

The approvers can be selected in the flow, and this can be a group of leads as well as one lead, and preferences can be set such as the ability to ensure all approvers respond, or the ability for the system to update the status based on the response of the first approver.

Django

Once the list of a maximum number of 10 reviews has been made available from the reviewers on the view feedback and review under each project in the reviews dashboard, a lead or the so-called approver is now able to log in and accept or reject the project. The reviews are in the read-only format, so that the approvers can have a holistic overview of the reviews and can make an informed decision whether or not to pursue the project. Both the approval and the rejection are done through a GET request method and assessing which reference of the two accept/reject button is sent to the method. Depending on the action button, the accepted or rejected projects database is reloaded and the current project is stored accordingly, as well as its display in the respective views.py Also, the system automatically sends an email through the send_mail() method to the submitter in order to inform them upon the final decision of the approver regarding their project proposal.

Data Storage - Databases


SharePoint

The lists can be exported with a query or without a query as a CSV file by clicking the export button that is built into the SharePoint lists. The accepted projects and project proposals are also stored in an Excel sheet, stored in the document library of the SharePoint site. The feedback data is also stored in the Excel sheets, which can be mined by companies to show feedback from different individuals.

Authentication. Permissions. Users


Django

There are three types of authenticated permission users, namely submitters, reviewers and approvers. As admin, we were able to log in the Django system's database and configure the Group's permissions.

The code we employed for writing the permissions bases on filtering the user.groups. By adding @is_reviewer to the top of a view method, this allowed only users which passed the boolean method to execute that view method and render the html. Therefore, submitters cannot access the review portal even if they input the url into their browser, as the view could not be accessed by them.

Deployment


SharePoint

The flows described above can be exported as a zip, which can be imported into another organisation's SharePoint site by importing the zip file and changing the connections. The connections can then be changed to the appropriate lists and users and deployed within an organisation's own SharePoint site. This is explained in the deployment video in the Appendicies.

The lists can be replicated within a SharePoint organisation site by running the site script provided. This will need to be done by the adminstrator of the organisation's Sharepoint. The site design can then be applied to the site and this creates the lists with the correct formating and styling, which can then be appended to pages and integrated into an existing SharePoint site.

Django

In order to deploy the Django web portal properly, open two bash terminals (one for the local server and one for the remote) - the commands will be alternative. Set up a LINODE account (we used Linode for our cloud computing and web hosting service). The IP address of the REMOTE website can be found in the Network tab under the IP addresses or via the SSH Access. We will use Ubuntu for the operating system needed to deploy the web portal. The series of instructions and their explanations are provided step-by-step in the Django Deployment Manual that can be found in the Appendices section.
