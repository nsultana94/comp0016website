Testing Strategy

[TODO]

Throughout the project we tried to adapt the agile technique of Test-Driven Development. This allowed us to test constantly after we implemented each feature, and allowed to us to ensure that each feature was fully functional and bug free before we continued working on another feature. We were able to adapt pair programming through the development process and so as a result, we were able to give feedback on each other's code and test for bugs. We would also have frequent code reviews of each other's code in order to make sure the code was seamless and bug free.

As we were trying to do TDD, this revealed to us many bugs as we developed, but they were easy to solve as it was easy to pinpoint the source of the bugs. This is because, we would only push to the main branch in the repo only if the code was functioning and so we knew that it was working up until that point. We also made great use of the github branch funtionality, so created branches for different versions of the code so we could preserve fully working code and implement new features on a new branch so we could test thoroughly before merging the branches.

Throughout the development process, we tested each feature thoroughly by testing the functionality as we went. For instance, for testing the submission portal, we tested the system's response to a submission with correct data formats, and then a submission with incorrect data formats, and ensured the results were as expected.

Unit and Integration Testing


When developing the SharePoint site, the most important aspect to test was the power automate flows. Therefore, after designing the flow, we tested each individual flow's functionality using Microsoft's inbuilt test functionality. This allowed us to manually trigger the flow by creating arbitrary data ourselves or automatically triggering the flow using a recent response or trigger. This allowed us test each flow separately, isolated from the entire system, and the system would show whether the flow failed or succeeded. It would show the point of failure clearly in the diagram, and so we were able to fix and restructure the design as we developed. By testing the flows individually, we could be confident that any source of bugs in functionality were not from the individual flows as we would be alerted by email if the flow failed.

Compatibility Testing


We developed the open-source django platform to be compatible for all screen sizes. The front-end was developed to use containers that were not fixed and fluid to the size of the screen. Initially, our side bar was fixed and our layout was fixed which causes display issues when changing the screen size from full screen. However, by changing the layout and containter types as well as adding an option to temporarily hide the navigation bar, this reduced the disarray from changing the screen size so all the key content was still shown and visible to the users.

We also tried to test on different operating systems and devices. However, given the nature of the pandemic, it was difficult to obtain a large variety of devices and operating systems as all our devices used MacOS and our family and friends also used similar devices, and this could only be tested after deployment. However, the testing did reveal some bugs in compatibility that should be addressed in the future.

Responsive Design Testing

We ran our Sharepoint site across multiple devices (smartphone and tablet) and there were no drawbacks regarding the key functionalities of the system. However, our system was not designed for small screens and therefore a user may find it more difficult to navigate the website due to the lack of screen space. We also ran our site across multiple browsers (Firefox, Chrome, Opera, Safari) and there were no issues.
