UI Design

User interface design

Sketches

Upon gathering users’ requirements, creating personas and envisaging scenarios, we designed several wireframe sketches that will smoothen the transition to a higher-fidelity, interactive prototype.W

We considered the possible idea of a timeline or a checklist system, so it was easy to track each stage of the application process and so the users are not overwhelmed by a large form of information. We iterated this process of multiple sketches in order to obtain various designs. The use of a timeline also made it clear to users what stage of the submission they were at and hence maintained the affordance of the program.

Submission Portal Sketches

The sketches above tried to encompass a user-friendly interaction which make the user experience easy to follow and a fluid process. We felt the submission portal was an integral part of the user experience.

Feedback System Sketches

We designed multiple sketches for the reviewing system considering a red-amber-green system for feedback as well as a point-based system. Users expressed they liked the red-amber-green method with a combination of point based as it allowed easy comparison of projects. This increased the visibility of the system which is an important design principle to consider, as it made it clear what the button's functionality was. It would also help the user experience by clearly highlighting what the RAG buttons were doing as well as the points and so it gave good feedback to the user of what functionalities they had as a user.


Prototype

Using our sketches, we produced a prototype in figma.com to illustrate the webapp framework and emphasise the usability and feasibility of the finished portal engine.​ We added a dashboard as most users expressed they wished to have a system that was easy to navigate. Non-distracting colours were used so it was clear and the main objective was the focus. This maintained the mapping in the design principles

The submission portal was designed using a timeline to separate the categories of the proposal form, which mimics the existing IXN project proposal form with additional sections such as the user’s motivations for submitting the project which some clients wished the system to implement.

On the other hand, we will be using Django for front-end and back-end for a public version of the portal engine, requested by the NHS. Their systems are not reliant on Microsoft and therefore would prefer a standalone software.
